Jakarta REST 5.0

CDI Integration

James R. Perkins

IBM

April 2026

Overview

  • Full CDI integration for dependency injection
    • Deprecates @Context in favor of @Inject
    • Backward compatible - existing code still works
  • Automatic discovery via CDI stereotypes
    • @Path, @Provider, @ApplicationPath are now stereotypes
    • No additional annotations needed for bean discovery
  • New @EntityParam annotation
    • Distinguishes request body from CDI beans in method parameters
  • Targeted for Jakarta EE 12

What's Deprecated

  • @Context annotation
    • Use @Inject for field and constructor injection
    • Resource method parameters do not need any annotation
  • @Suspended annotation
    • AsyncResponse identified by type alone
    • Consistent with CDI type-based injection
  • ContextResolver interface
    • Replaced by CDI producers
    • Use standard CDI patterns instead

All deprecated APIs still work - no breaking changes!

Remove the @Context from constructor and method parameters.

Field Injection - Before & After

Jakarta REST 4.0


@Path("/greet1")
public class SampleResource {

    @Context
    private UriInfo uriInfo;

    @HeaderParam("who")
    private String who;

    private final GreetBean bean;
}

Jakarta REST 5.0


@Path("/greet1")  // bean-defining annotation
@RequestScoped    // implied by default
public class SampleResource {

    @Inject
    private UriInfo uriInfo;

    @Inject
    @HeaderParam("who")
    private String who;

    private final GreetBean bean;
}

Constructor Injection


@Path("/greet1")
public class SampleResource {

    private final GreetBean bean;
    private final String lang;

    @Inject
    public SampleResource(GreetBean bean,
                          @QueryParam("lang") String lang) {
        this.bean = bean;
        this.lang = lang;
    }
}
  • Constructor must be annotated with @Inject
  • Parameter annotations work as CDI qualifiers
  • Standard CDI constructor injection

Constructor Injection - Before & After

Jakarta REST 4.0


@Path("/headers")
public class SampleResource {

    private final HttpHeaders headers;
    private final String lang;

    public SampleResource(@Context HttpHeaders headers,
                          @QueryParam("lang") String lang) {
        this.headers = headers;
        this.lang = lang;
    }
}

Jakarta REST 5.0


@Path("/headers")
public class SampleResource {

    private final HttpHeaders headers;
    private final String lang;

    @Inject
    public SampleResource(HttpHeaders headers,
                          @QueryParam("lang") String lang) {
        this.headers = headers;
        this.lang = lang;
    }
}

Resource Method Parameters

  • No annotation needed on the method itself
    • The resource methods will be invoked with CDI method invokers
    • Don't use @Inject on methods - it means something different in CDI!
  • Parameter annotations act as CDI qualifiers
    • @PathParam, @QueryParam, etc. qualify the injection point
    • CDI producers resolve values from the request context
  • Inject arbitrary CDI beans as parameters
    • Any CDI-managed bean can be a parameter
    • Validator services, business beans, etc.
  • Critical point: DO NOT put @Inject on the method itself
  • In CDI, @Inject on a method means "initializer method" - called once during bean construction
  • Resource methods are invoked by CDI for each request, not as initializers
  • Parameters use annotations as CDI qualifiers - this is the new pattern

The @EntityParam Requirement

Why it's needed:


@POST
@Consumes(MediaType.APPLICATION_JSON)
public Response create(@EntityParam Customer customer,  // Request body
                       CustomerValidator validator) {   // CDI bean
    // Without @EntityParam, how would we know customer
    // is the request body vs. another CDI bean?
}
  • Distinguishes request body from CDI beans
  • Deserialized using MessageBodyReader
  • Only one @EntityParam per method
  • This is NEW in 5.0 - addresses ambiguity problem
  • Without it: if Customer is a CDI bean, is it the request body or injected bean?
  • @EntityParam makes it explicit: this parameter is the deserialized request body
  • This will be the biggest issue we have to address with compatibility.
  • Only one @EntityParam per method allowed

Resource Methods - Before & After

Jakarta REST 4.0


@POST
@Path("/{id}")
public Response createOrder(@Context HttpHeaders headers,
                            @PathParam("id") long id,
                            Order order) {
    // Process order
}

Jakarta REST 5.0


@POST
@Path("/{id}")
public Response createOrder(HttpHeaders headers,
                            @PathParam("id") long id,
                            @EntityParam Order order) {
    // Process order
}

Example - Server-Sent Events

Jakarta REST 4.0


@GET
@Path("sse")
@Produces(MediaType.SERVER_SENT_EVENTS)
public void getMessage(
    @HeaderParam("who") String who,
    @Context Sse sse,
    @Context SseEventSink sseEventSink) {
    sseEventSink.send(
        sse.newEvent(
            bean.getMessage(lang) + " " + who));
}

Jakarta REST 5.0


@GET
@Path("sse")
@Produces(MediaType.SERVER_SENT_EVENTS)
public void getMessage(
    @HeaderParam("who") String who,
    Sse sse,  // Type-based injection
    SseEventSink sseEventSink) { // Type based
    sseEventSink.send(
        sse.newEvent(
            bean.getMessage(lang) + " " + who));
}

Automatic Discovery with Stereotypes

Jakarta REST annotations are now CDI stereotypes:

  • @Path → @RequestScoped + @Stereotype
    • Resources are request-scoped by default
    • Automatically discovered by CDI
  • @Provider → @ApplicationScoped + @Stereotype
    • Providers are application-scoped
  • @ApplicationPath → @ApplicationScoped + @Stereotype

Default scopes can be overridden:


@Path("/admin")
@ApplicationScoped  // Override default @RequestScoped
public class AdminResource { }
  • These annotations are now stereotypes - means they're bean-defining
  • @Path resources: @RequestScoped by default (new instance per request)
  • @Provider: @ApplicationScoped (singleton)
  • Can override with explicit scope annotation, but use @ApplicationScoped resources carefully
  • CDI discovers them automatically

Application.getClasses() with CDI

Understanding the separation:

  • CDI manages bean creation
    • All @Path and @Provider classes become CDI beans
    • Independent of Application.getClasses()
  • getClasses() controls request routing
    • Determines which resources handle requests
    • Doesn't control CDI bean creation
  • getSingletons() instances are NOT CDI-managed
    • Plain objects, no CDI injection

Example: A @Path resource not in getClasses() is still a CDI bean, just not routable.

  • Common confusion: getClasses() vs CDI bean creation are SEPARATE
  • CDI creates ALL @Path/@Provider classes as beans regardless of getClasses()
  • getClasses() only controls ROUTING - which resources handle requests
  • getSingletons() instances are NOT CDI-managed - plain objects
  • A resource not in getClasses() is still a CDI bean, just not routable

Migration Guide

Quick migration checklist:

  • Fields & Constructors: Replace @Context with @Inject
  • Entity Parameters: Add @EntityParam to request body parameters
  • AsyncResponse: Remove @Suspended annotation (type alone is enough)
  • Optional: Explicitly set CDI scopes if needed

Backward compatibility:

  • Old @Context code still works (deprecated for removal)
  • No immediate code changes required
  • Migrate at your own pace
  • The message request body will be the biggest issue we need to solve.
